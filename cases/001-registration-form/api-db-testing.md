Метод: отправка запросов с намеренно изменёнными данными для проверки граничных случаев.

| # | Сценарий | Тело запроса | Ожидаемый статус | Фактический | Вывод |

## Проверка базы данных (DBeaver)

**Подключение:** 127.0.0.1:5433, база `sortula`, пользователь `sortula_user`

```sql
SELECT id, email, email_verified, hashed_password, created_at
FROM users
ORDER BY created_at DESC
LIMIT 5;
```

**Результаты:**
- Запись пользователя найдена в таблице `users` сразу после регистрации через API
- Поле `hashed_password` имеет тип `varchar(255)` — хранится bcrypt-хеш, не открытый текст
- Значение `hashed_password = NULL` у только что зарегистрированного пользователя — пароль не устанавливается при регистрации
- Поле `email_verified = false` до прохождения верификации

## Ключевая находка: пароль при регистрации намеренно не принимается

**Наблюдение:** при передаче поля `password` в теле запроса `POST /v1/auth/register` API возвращает 201, но `has_password: false`. Пароль не сохраняется.

**Причина (подтверждена анализом кода):** схема `UserCreate` в backend намеренно не содержит поле `password`. FastAPI молча игнорирует неизвестные поля. Пароль устанавливается отдельным эндпоинтом `POST /v1/auth/set-password`.

**Это намеренная архитектура, не баг.** Sortula поддерживает несколько способов входа:
- Telegram (без пароля)
- Magic link (одноразовая ссылка на email)
- Email + пароль (пароль задаётся через `/v1/auth/set-password` после верификации email)

**Влияние на ранее открытую зону:** в итоговом отчёте (Спринт 4) была отмечена открытая зона: *«Механизм установки пароля при email-регистрации — зависит от разгадки BUG-02»*. Теперь механизм установлен: пароль устанавливается через отдельный эндпоинт, а не через письмо.

## Закрытие открытой зоны по BUG-02

BUG-02 в исходном отчёте описывал недоставку письма-подтверждения и был предварительно повышен до Blocker с примечанием: *«если пароль устанавливается именно через это письмо»*.

**Факты, собранные в Спринте 5:**

1. `hashed_password = NULL` в БД — пароль через письмо не устанавливается
2. `POST /v1/auth/login` возвращает `403 Forbidden «Email не подтверждён»` — API корректно требует верификации, это не баг
3. Код `UserCreate` не принимает `password` — осознанное архитектурное решение

**Вывод:** severity BUG-02 остаётся **Major** (не Blocker). Недоставка писем блокирует флоу email-верификации, что не позволяет войти в аккаунт через email-флоу, — но механизм установки пароля тут ни при чём. Уточнение сделано на основе фактов из кода и БД, а не предположений.
