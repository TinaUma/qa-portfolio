# Кейс: Тестирование формы регистрации (staging.sortula.ru)

**Что сделано:** многослойное тестирование флоу регистрации и авторизации — от UI до API и базы данных.

**Слои тестирования:**
- **UI (Спринты 1–4):** декомпозиция требований, тест-анализ (классы эквивалентности, граничные значения), чек-лист из 25 пунктов, ручное тестирование email-флоу и Telegram-флоу, регрессия, exploratory-сессия по флоу восстановления пароля
- **API + БД (Спринт 5):** мутационное тестирование `POST /v1/auth/register` через Postman (5 сценариев), прямая верификация данных в PostgreSQL через DBeaver, закрытие открытой зоны по механизму установки пароля

**Найдено дефектов:** 7 (UI-слой), из них Critical: 1, Major: 3, Minor: 3.

**Ключевые находки:**
- Критический разрыв в пользовательском флоу авторизации через Telegram (сервис технически работает, но пользователь не может пройти его без посторонней подсказки)
- Проблема с недоставкой писем-подтверждений (регистрация и восстановление пароля), severity уточнён до Major (не Blocker) — подтверждено через API и БД в Спринте 5
- Систематическая ошибка в сообщениях валидации для кириллических email
- Утечка технического ответа API вместо человекочитаемого сообщения об ошибке

**Инструменты:** ручное тестирование, Chrome DevTools (Network), Postman Desktop, DBeaver, SQL (SELECT/WHERE), декомпозиция входных данных, exploratory-тестирование по чартеру.

**Методология:** каждая находка проверена на воспроизводимость минимум дважды. Открытые зоны из UI-слоя закрыты в API-слое через прямые запросы к коду и БД — баги фиксируются фактами, не предположениями.

**Вывод:** не готово к релизу — критический дефект в Telegram-флоу и системная недоставка писем блокируют базовые пользовательские сценарии.
