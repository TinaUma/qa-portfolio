# Кейс: Тестирование формы регистрации

**Объект тестирования:** форма регистрации на staging.sortula.ru  
**Дата:** 08.07.2026  
**Тестировщик:** Тина Умашева  
**Тип тестирования:** ручное, функциональное, UX

---

## Что сделано

- Декомпозиция формы регистрации: 7 объектов тестирования, выявление серых зон и допущений
- Тест-анализ с применением классов эквивалентности и граничных значений
- Чек-лист из 25 пунктов с результатами прогона
- Сквозное тестирование двух флоу регистрации: email и Telegram
- Анализ сетевых запросов через Chrome DevTools (Network tab)
- Оформление 7 баг-репортов с severity, priority и шагами воспроизведения

## Результат

**Найдено дефектов: 7** — из них Critical: 1, Major: 3, Minor: 3

| # | Баг | Severity |
|---|---|---|
| BUG-01 | Некорректное сообщение об ошибке для кириллического email | Minor |
| BUG-02 | Письмо-подтверждение регистрации не доставляется | Major |
| BUG-03 | Отсутствует валидация максимальной длины email (RFC 5321) | Minor |
| BUG-04 | Сырой JSON вместо читаемого сообщения об ошибке (форма логина) | Major |
| BUG-05 | Ссылки на юридические документы открываются в текущей вкладке | Minor |
| BUG-06 | Кнопка «START BOT» не инициирует /start автоматически | Major |
| BUG-07 | Авторизация через Telegram: пользователь не понимает, что делать со ссылкой | **Critical** |

## Ключевые находки

- **Критический разрыв в Telegram-флоу:** сервис технически работает, но пользователь не может пройти его без посторонней подсказки — подтверждено экспериментально
- **Потенциально блокирующая проблема:** письма-подтверждения не доставляются ни на один из 4 тестовых адресов; если пароль устанавливается именно через письмо — severity BUG-02 следует повысить до Blocker
- **Систематическая ошибка в валидации:** некорректный текст для кириллических email подтверждён на двух примерах — это не единичный случай
- **Утечка архитектурных деталей:** необработанный Pydantic/FastAPI ответ отображается напрямую пользователю

## Методология

Каждая находка проверялась на воспроизводимость минимум дважды перед фиксацией как баг. Например, гипотеза про символ «+» в email была опровергнута дополнительным тестом с точкой — это привело к более точному пониманию проблемы через уточнение у команды.
