# Итоговый отчёт о тестировании: Форма регистрации и авторизации (staging.sortula.ru)

**Тестировщик:** Тина Юмашева
**Период:** 08.07.2026–13.07.2026 (Спринты 1–7)
**Окружение:** staging.sortula.ru + localhost:8001 (dev) | Chrome DevTools | Postman | DBeaver | Android Studio

---

## 1. Объём прогона

Сквозной кейс — один флоу (регистрация/авторизация), протестированный на всех уровнях стека последовательно:

- **UI (Спринты 1–4):** тест-анализ, чек-лист 25 пунктов, ручное тестирование email- и Telegram-флоу, регрессия, exploratory-сессия по восстановлению пароля
- **API + БД (Спринт 5):** мутационное тестирование `POST /v1/auth/register` (5 сценариев), SQL-верификация в PostgreSQL через DBeaver
- **Логи (Спринт 6):** расследование через `docker compose logs` + `grep`, root cause BUG-02 установлен на уровне кода и конфигурации
- **Мобильный веб + PWA (Спринт 7, часть A):** Chrome DevTools Device Mode (iPhone SE), PWA-аудит (Manifest/Service Worker/Offline), Lighthouse
- **Android-полигон (Спринт 7, часть B):** Android Studio, AVD Manager, Logcat — навык отработан отдельно, у Sortula нет нативного APK (подтверждено `vision.md`)

| BUG-08 | Валидация формата пароля на логине до проверки хеша (архитектурная ошибка) | API/Код | Major | Открыт (технический долг, не активный инцидент) |

**Итого:** 14 дефектов — 2 Critical/Blocker, 5 Major, 7 Minor — плюс 2 задокументированных архитектурных решения (не баги): офлайн-контент не кешируется (приватность), `/bookmarks` закрыт от индексации (приватность).

**Связь между слоями (пример сквозного расследования):** один и тот же тестовый ввод (`TestPass999`, пароль без спецсимвола на логине) вскрыл два разных дефекта в разные спринты — Спринт 4 обнаружил симптом на UI (BUG-04, сырой JSON), Спринт 7 нашёл архитектурную первопричину в коде (BUG-08, валидация формата не к месту). Это иллюстрирует, зачем тестировать один флоу на разных уровнях, а не только через интерфейс.

## 3. Что осталось не протестировано (актуальные открытые зоны)

- Поведение формы восстановления пароля при невалидном/несуществующем email — не проверено ни на одном из слоёв
- Кроссбраузерность (Firefox, Safari) — сознательно отложена с Спринта 4 (приоритет отдан аналитике трафика), мобильная адаптивность при этом закрыта в Спринте 7
- Распространяется ли проблема с SMTP (BUG-02) на другие виды уведомлений, помимо auth-писем — не проверено, не обобщать без данных

*(Закрыто с прошлой версии отчёта: механизм установки пароля при регистрации — разгадан в Спринте 5; мобильная адаптивность — протестирована в Спринте 7)*

## 4. Вывод о готовности к релизу

**Не готово к релизу.**

Обоснование усилилось по сравнению с первичной оценкой (Спринт 4): помимо критического разрыва в Telegram-флоу (BUG-07) и полностью неработающей доставки писем (BUG-02, теперь подтверждённо Critical/Blocker для email-флоу), обнаружены системные проблемы производительности на мобильном (BUG-09, LCP в 3 раза выше нормы) и архитектурная уязвимость в логике логина (BUG-08), которая может заблокировать пользователей в будущем без видимой причины. Ни один дефект не был устранён между первичным обнаружением и последующими проверками на всех слоях.

**Рекомендация по приоритету исправления:**
1. BUG-02 — настройка SMTP (DevOps, не требует изменений кода)
2. BUG-07 — переработка UX Telegram-флоу
3. BUG-08 — убрать валидацию формата пароля из `UserLogin`
4. BUG-04, BUG-06, BUG-09 — далее по значимости

## 5. Методологическое резюме

Каждая находка проверена минимум на двух независимых источниках доказательств (UI-наблюдение + API-ответ, лог + код, БД-значение + структура таблицы) — баги фиксируются фактами, не предположениями. Оценки severity уточнялись по мере поступления новых данных (пример: BUG-02 прошёл путь Major → Major(уточнён) → Critical/Blocker по мере расследования) — это отражает добросовестный процесс, а не непоследовательность.
