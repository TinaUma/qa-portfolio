## BUG-01: Некорректное сообщение об ошибке для кириллического email

**Severity:** Minor
**Priority:** Low

**Окружение:** staging.sortula.ru, Chrome, форма регистрации

**Шаги воспроизведения:**
1. Открыть форму регистрации
2. Ввести email с кириллицей, например `тест@пример.рф`
3. Наблюдать сообщение об ошибке

**Ожидаемый результат:** Понятное сообщение вида «Email должен содержать только латинские символы»

**Фактический результат:** Сообщение указывает на конкретный символ первой буквы слова: «Часть адреса до символа "@" не должна содержать символ "т"». Паттерн подтверждён на втором примере (`пример@пример.рф` → «...символ "п"») — ошибка систематическая

## BUG-02: Письмо-подтверждение регистрации не доставляется — root cause установлен

**Severity:** Critical/Blocker (для email-флоу регистрации)
**Priority:** High
**Категория:** Configuration/Environment Issue — не дефект кода, исправление на стороне DevOps

**История оценки severity:**
- Спринт 4 (первичное обнаружение): Major, потенциально Blocker — зависело от того, устанавливается ли пароль через письмо
- Спринт 5 (API+БД): уточнено до Major (не Blocker) — установлено, что пароль устанавливается отдельным эндпоинтом, письмо не завязано на пароль
- Спринт 6 (логи+код): **финализировано до Critical/Blocker** — установлено, что SMTP не настроен вообще, письма не уходят ни при каких условиях в текущей конфигурации staging — весь email-флоу верификации непроходим для любого пользователя

**Окружение:** staging.sortula.ru (root cause подтверждён на локальном dev, идентичная конфигурация)

**Шаги воспроизведения:**
1. Зарегистрироваться с валидным email
2. Проверить почтовый ящик — письмо не приходит
3. (Спринт 6) Проверить логи backend: `docker compose logs -f backend`

**Фактический результат (Спринт 6):** Backend формирует письмо полностью корректно — токен создаётся, письмо рендерится с брендингом — но в dev-режиме сохраняется в файл вместо отправки (`SMTP_USERNAME` пустой в `.env`). На staging — идентичная причина.

**Доказательная цепочка:**
```
docker compose logs -f backend
→ [DEV MODE] Email сохранён: /app/temp/emails/....html
grep "set-password?token" по файлу письма → токен извлечён
POST /set-password + токен → 200 OK, email_verified=true
POST /login → 200 OK + JWT
```

Код работает от начала до конца безупречно — единственная точка отказа: отсутствие SMTP-конфигурации в окружении.

**Вывод:** это не баг кода, а операционный пробел. Тем не менее для реального пользователя на этой конфигурации результат идентичен блокеру — он физически не может завершить email-регистрацию.

**Скриншоты:** [evidence/bug-02-email-not-delivered-network-201.PNG](evidence/bug-02-email-not-delivered-network-201.PNG), полная цепочка — [api-db-testing.md](api-db-testing.md#спринт-6-расследование-через-логи-12072026)

## BUG-03: Отсутствует валидация максимальной длины email

**Severity:** Minor
**Priority:** Low

**Окружение:** staging.sortula.ru, Chrome

**Шаги воспроизведения:**
1. Открыть форму регистрации
2. Ввести email со 149 символами до символа «@» (без пробелов и недопустимых символов), домен gmail.com
3. Нажать «Зарегистрироваться»

**Ожидаемый результат:** По стандарту RFC 5321 максимальная длина local part — 64 символа. При превышении система должна показать ошибку о длине

**Фактический результат:** Email принят без ошибки, показано «Проверьте почту» — валидация по длине отсутствует

**Скриншот:** [evidence/bug-03-long-email-149chars.PNG](evidence/bug-03-long-email-149chars.PNG)

## BUG-04: Сырой JSON вместо читаемого сообщения об ошибке (форма логина)

**Severity:** Major
**Priority:** Medium

**Окружение:** staging.sortula.ru, Chrome, форма входа (/login)

**Шаги воспроизведения:**
1. Перейти на форму логина через «Уже есть аккаунт? Вход»
2. Ввести email и пароль без спецсимвола (например, `TestPass999`)
3. Нажать «Войти»

**Ожидаемый результат:** Понятное, отформатированное сообщение об ошибке для пользователя

**Фактический результат:** Показан необработанный ответ backend API:
```
[{"type":"value_error","loc":["body","password"],"msg":"Value error, Пароль должен содержать хотя бы один спецсимвол (!@#$%^&* и т.д.)","input":"TestPass999","ctx":{"error":{}}}]
```
Сообщение технически корректно по содержанию, но не адаптировано для пользователя и раскрывает детали архитектуры (похоже на Pydantic/FastAPI validation error)

**Скриншот:** [evidence/bug-04-raw-json-login-error.PNG](evidence/bug-04-raw-json-login-error.PNG)

**Priority:** High

**Окружение:** staging.sortula.ru → t.me/@SortulaStagingManagerBot, Telegram Desktop

**Шаги воспроизведения:**

## BUG-07: Авторизация через Telegram — пользователь не понимает, что делать со ссылкой

**Severity:** Critical
**Priority:** High

**Окружение:** staging.sortula.ru, Telegram Desktop

**Шаги воспроизведения:**
1. Вручную ввести `/start` в чат с ботом (обходной путь для BUG-06)
2. Получить сообщение «Вход в Sortula» со ссылкой вида `https://staging.sortula.ru/auth/tg?token=...`
3. Кликнуть на ссылку в Telegram

**Ожидаемый результат:** Понятная инструкция о дальнейших действиях (например: «Перейдите по ссылке в браузере для завершения входа»)

**Фактический результат:** При клике на ссылку появляется только уведомление «Текст скопирован в буфер обмена» — без пояснения, что нужно вставить ссылку в браузер. Технически авторизация работает: вставка ссылки в адресную строку браузера завершает вход успешно — но обычный пользователь не догадается об этом шаге без посторонней подсказки

**Связь с другим багом:** в сочетании с BUG-06 (кнопка START BOT не отправляет /start автоматически) весь Telegram-флоу регистрации практически непроходим для обычного пользователя без инструкции

**Скриншот:** механизм подтверждён экспериментально — вставка ссылки в адресную строку браузера привела к успешной авторизации

## BUG-08: Валидация формата пароля срабатывает при логине до проверки хеша (архитектурная ошибка)

**Severity:** Major
**Priority:** Medium
**Связан с:** BUG-04 (тот же тестовый ввод `TestPass999` вскрыл два разных бага в разных спринтах — Sprint 4 показал симптом на UI, Sprint 7 нашёл архитектурную причину в коде)

**Окружение:** localhost:8001 (dev), Postman

**Шаги воспроизведения:**
1. `POST /v1/auth/login` с телом `{"email": "...", "password": "TestPass999"}` (пароль без спецсимвола)
2. Наблюдать ответ

**Ожидаемый результат:** Логин должен только сравнивать пароль с хешем в БД — не проверять формат введённой строки

**Фактический результат:** `422 Unprocessable Content` — валидация формата (наличие спецсимвола) срабатывает до обращения к БД. Подтверждено кодом: `UserLogin` и `UserSetPassword` используют один и тот же `Password`-валидатор (`backend/app/schemas/user.py`), тест `test_rejects_no_special_char` проверяет это на `UserLogin` напрямую

**Почему это баг, а не просто строгая валидация:** логин работает с уже существующим паролем пользователя — проверка его "формата" бессмысленна и потенциально опасна: если требования к паролю когда-либо смягчатся на `set-password`, пользователи с "мягкими" паролями будут заблокированы на входе навсегда, даже вводя верный пароль

**На данный момент реальных пользователей не задевает** (все пароли в БД гарантированно проходят одинаковый валидатор при создании) — это архитектурный технический долг, не активный инцидент

**Скриншот:** [mobile-testing.md](mobile-testing.md#ключевая-находка-1--архитектурный-смелл-на-логине)

## BUG-09: LCP 7.8s на мобильном viewport (норма 2.5s)

**Severity:** Major
**Priority:** Medium

**Окружение:** staging.sortula.ru, Chrome DevTools Device Mode, iPhone SE (375×667)

**Шаги воспроизведения:**
1. Открыть `/bookmarks` в Device Mode на iPhone SE
2. Запустить Lighthouse-аудит (вкладка Performance)

**Ожидаемый результат:** LCP ≤ 2.5s (порог Google Core Web Vitals)

**Фактический результат:** LCP = 7.8s — в 3 раза выше нормы. Причины по Lighthouse Insights: неиспользуемый JS (522 КБ), неоптимизированные изображения (224 КБ экономии)

**Влияние:** пользователь на мобильном интернете ждёт 7+ секунд, пока загрузится список закладок — воспринимается как "сайт тормозит"

**Скриншот:** [evidence/sprint-07/block-a/lighthouse repport.PNG](evidence/sprint-07/block-a/lighthouse%20repport.PNG)
