# Отчёт о нагрузочном тестировании

**Эндпоинт:** `POST /v1/auth/register`
**Продукт:** Sortula — AI-powered bookmarking SaaS
**Дата:** 23.07.2026
**Тестировщик:** Тина Юмашева
**Инструмент:** Locust 2.46.1 (Python)

---

## 1. Цель тестирования

Проверить поведение эндпоинта регистрации под нагрузкой 20 одновременных пользователей:
- Остаётся ли время отклика в допустимых пределах?
- Есть ли деградация производительности при росте нагрузки?
- Как система ведёт себя при высокочастотных запросах с одного IP?

---

## 4. Ключевые находки

### Находка 1: Rate limiter активен и работает корректно

**Severity:** Informational (ожидаемое поведение)

При нагрузке ~10 RPS с одного IP эндпоинт регистрации возвращает `429 Too Many Requests` с cooldown 59 минут. На графике момент активации виден как мгновенное падение времени ответа с ~1500 ms до ~0 ms — сервер отдаёт 429 без обработки запроса.

Это корректная защита от спам-регистраций. Для нагрузочного тестирования staging необходимо исключение тестового IP из rate limit.

### Находка 2: Производительность FastAPI-бэкенда отличная

**Severity:** Positive finding

Реальное время обработки запроса сервером — **3 ms** (p50, localhost). p95 = 5 ms. Это значительно лучше индустриальной нормы (< 200 ms для p50, < 500 ms для p95).

Сетевая задержка до staging составляет ~21 ms — нормальный показатель для интернет-соединения.

### Находка 3: Единичный пик 822 ms (localhost, сценарий A)

**Severity:** Low

Максимальный отклик 822 ms зафиксирован в начале прогона — вероятно, холодный старт Docker-контейнера (инициализация соединений с БД). В установившемся режиме пиков нет.

---

## 5. Выводы

Система **готова к нормальной нагрузке** по показателям производительности:
- p50 = 3 ms, p95 = 5 ms — значительно лучше нормы
- 0% failures при корректной конфигурации (без rate limit)
- RPS 9.9 обрабатывается стабильно без деградации

Rate limiter **работает как задумано** и защищает эндпоинт регистрации от злоупотреблений.

## 6. Рекомендации

| # | Рекомендация | Приоритет |
|---|---|---|
| 1 | Для нагрузочного тестирования staging: добавить whitelist тестовых IP или отдельный конфиг rate limit для QA-окружения | Medium |
| 2 | Задокументировать порог rate limiter (запросов/мин с IP) для команды разработки | Low |
| 3 | Провести нагрузочный тест с большим числом пользователей (50–100) для определения реального потолка производительности | Low |
